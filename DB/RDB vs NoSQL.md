 # RDB란?
- RDB(Relational Database)는 데이터를 행(Row)와 열(Column)이 있는 표 형태로 저장하고 엑셑(Excel)과 유사한 형태입니다
- 관계형 DB는 가장 많이 사용되는 데이터베이스의 유형으로, Schema(스키마)라는 틀에 맞춰 데이터를 저장합니다
- 테이블에 들어갈 데이터의 자료형과 제약사항(constraint)을 모두 명시합니다
- 대표적으로 MYSQL, MariaDB, PostgreSQL, SQLite, OracleDB 등이 있습니다
### 강점
- 데이터의 무결성
     - 개체 무결성: 모든 테이블이 PK를 가져야하고 PK는 중복을 허용하지 않아야 하고 빈값을 사용하지 않습니다
     - 참조 무결성: FK가 참조하는 테이블의 PX와 일치해야하고 이를 통해 데이터 간의 일관성을 보장합니다
     - 범위 무결성: Column의 값의 데이터의 범위에 대한 제약 조건을 보장한다는 원칙으로 Schema를 통해 정의한 데이터 타입, 길이, 범위 등의 제약이 보장됩니다.
- 명확한 구조 보장: Schema에 맞춰 테이블을 선언하고 해당 테이블에 데이터를 저장합니다
- ACID는 사고가 발생해도 데이터베이스에 저장되어 있는 유효성을 보장합니다
### 약점
- 경직된 Schema
- 과도한 join으로 인한 성능 하락
- scale-out이 쉽지 않음
# NoSQL이란?
- NoSQL은 한 가지 종류의 DB를 뜻하는 것이 아닌 Document, Key-Value, Graph와 같이 SQL이 아닌 모든 DB들의 그룹을 이야기합니다
- RDB와 달리 테이블 간의 관계를 정의하지 않고 그렇기 때문에 join 연산이 블가능합니다
### 강점
- 유연한 스키마
- 읽기 속도가 매우 빠르다 (중복 허용)
- Scale-out에 용이
### 약점
- 대부분 Transaction을 지원X
- update의 속도가 느림
- Schema가 엄격하지 않기 때문에 데이터 구조 결정이 어렵고 어플리케이션 단에서 고려해야함
# 둘의 차이점
NoSQL은 RDB의 못하는 부분을 보완하기 위해 등장한 상호 보완적인 존재입니다
### RDB 이상적인 상황
- 높은 수준의 보안과 데이터 무결성이 필요
- 데이터를 저장하는 구조가 자주 바뀌지 않는 상황
- 수평적으로 확장할 필요가 없는 상황
- 자산이나 계좌와 같이 트렉젝션을 지원해야 할 떄
### NoSQL 이상적인 상황
- 높은 수준의 보안이나 데이터 무결성이 딱히 필요하지 않을 때
- 데이터의 구조가 명확하지 않아 자주 바뀔 것으로 예상되는 상황
- 유연한 동적 스키마가 필요할 때
- 수평적 확장이 필요할 때
- 데이터베이스에 저장되어 있는 데이터를 자주 변경하지 않을 때

# Broken Access Control
접근통제는 **사용자가 허용된 권한 범위를 벗어나 행동하지 못하도록 정책을 강제하는 것**입니다

이게 실패하면 다음 같은 문제가 발생합니다

**권한 없는 정보 열람**, **전체 데이터의 수정/삭제** 또는 **사용자 권한을 넘어서는 비즈니스 기능 수행**

### 순위

1. **CWE-200(권한 없는 사용자에게 민감 정보 노출)**
2. **CWE-201(전송 데이터로 인한 민감 정보 노출)**
3. **CWE-918(SSRF)**
4. **CWE-352(CSRF)**

### Broken Access Control 취약점

- **최소 권한 원칙 위반**
기본은 거부여야 하는데 특정 역할/권한/사용자에게만 허용돼야 할 기능이 아무한테 열려있는 경우
- **접근 통제 검사 우회**
URL 변경, 파라미터 변조, 강제 브라우징, 내부 애플리케이션 상테 조작, HTML 수정, 또는 API 요청을 바꾸는 공격 도구로 권한 검사를 우회하는 경우
- **IDOR(Insecure Direct Object Reference)**
다른 사람 계정의 고유 식별자를 넣어서 그 계정 정보를 조회하거나 수정할 수 있는 경우
- **API 메서드별 접근 통제 누락**
POST, PUT 같은 요청에 대한 접근 통제가 빠져있는 API
- **권한 상승**
로그인하지 않은 상태에서 사용자처럼 행동하거나 일반 사용자로 로그인 했는데 관리자 권한까지 얻은 경우
- **메타데이터 변조**
JWT 액세스 토큰 재사용/변조, 쿠키나 hidden field 조작으로 권한을 올리거나 JWT 무효화 처리를 악용하는 경우
- **CORS 설정 오류**
인증되지 않았거나 신뢰할 수 없는 출처에서 API 접근이 가능해진 경우
- **강제 브라우징**
비인증 사용자가 인증 페이지 URL을 추측해서 접근하거나 일반 사용자가 관리자 페이지 URL을 추측해 접근하는 경우

### **방지 방법**

접근 통제는 **신뢰할 수 있는 서버 측 코드나 서버리스 API에서 구현될 때만 효과적**입니다

공격자는 클라이언트 쪽의 접근 통제 검사나 메타데이터를 바꿀 수 있기 때문입니다

- 공개 리소스를 제외하고 기본 거부로 설계합니다
- 접근 통제 메커니즘을 한 번 구현하고 전반에 재사용합니다 CORS사용도 최소화합니다
- 사용자가 아무 레코드나 생성, 조회, 삭제 등을 못하게 하고 레코드 소유권을 강제해야 합니다
- 비즈니스 로직상 필요한 제한도 도메인 모델에서 강제해야 합니다
- 웹 서버의 디렉터리 목록 보기를 끄고 .git 같은 파일 메타데이터나 백업 파일이 웹 루트에 노출되지 않게 합니다
- 접근 통제 실패 로그를 남기고 반복 실패 같은 경우 관리자가 알 수 있게 합니다
- 자동화 공격 피해를 줄이기 위해 API와 컨트롤러에 rate limit를 둡니다
- 상태 기반 세션 ID는 로그아웃 후 서버에서 무효화합니다 짧은 수명을 가져야 합니다
- 개발자와 QA는 단위 테스트와 통합 테스트에 **기능적 접근 통제 검증**을 포함해야 합니다

# Software_Supply_Chain_Failures
소프트웨어 공급망 실패는 소프트웨어를 빌드하고, 배포, 업데이트 하는 과정에서 발생하는 붕괴나 침해를 말합니다

.보통은 시스템이 의존하는 서드파티 코드, 도구, 기타 의존성에 있는 취약점이나 악의적 변경 때문에 발생합니다

### 취약점

- 사용하는 모든 컴포넌트의 버전을 정확히 추적하지 않는 경우
클라이언트/서버 모두 포함이고 직접 쓰는 의존성 뿐만 아니라 전이 의존성까지 포함됩니다
- 소프트웨어가 취약하거나, 지원 종료됐거나 오래된 경우
여기에는 OS, 웹서버, DBMS, API, 런타임, 라이브러리 모두 다 포함됩니다
- 정기적으로 취약점을 스캔하지 않고 사용하는 컴포넌트 관련 보안공지도 구독하지 않는 경우
- 공급망 내부의 변경 사항을 추적하는 변경 관리 프로세스가 없는 경우
IDE, IDE 확장, 저장소 변경, 샌드박스 등 바뀐 건 전부 기록해야합니다
- 공급망의 모든 부분을 하드닝하지 않았고 특히 접근 통제와 최소 권한에 쓰지 않은 경우
- 공급망 시스템에 직무 분리가 없는 경우
한 사람이 코드를 작성하고 검토도 없이 바로 프로덕션까지 올릴 수 있으면 위험합니다
- 기술 스택 어디에서든 신뢰할 수 없는 출처의 컴포넌트를 사용하거나 그런 컴포넌트가 운영 환경에 영향을 줄 수 있는 경우
- 플랫폼, 프레임워크 의존성을 위험 기반으로 제때 패치 업그레이드 하지 않는 경우
- 개발자가 업데이트되거나 패치된 라이브러리의 호환성 테스트를 하지 않는 경우
- 시스템 전반의 설정을 안전하게 보호하지 않는 경우
- CI/CD 파이프라인 보안 수준이 그것이 빌드 배포하는 시스템보다 더 약한 경우
특히 파이프라인이 복잡할수록 더 위험합니다

### 해결법

- 전체 소프트웨어의 SBOM을 중앙에서 생성 관리합니다
- 직접 의존성뿐 아니라 전이 의존성까지 추적합니다
- 사용하지 않는 의존성, 불필요한 기능, 컴포넌트, 파일, 문서를 제거해서 공격 표면을 줄입니다
- 클라이언트와 서버 양쪽 컴포넌트의 버전과 의존성을 지속적으로 인벤토리 관리합니다
- CVE, NVD, OSV 같은 소스를 지속적으로 모니터링 해서 사용하는 컴포넌트의 취약점을 추적합니다
- 컴포넌트는 공식적으로 신뢰할 수 있는 출처에서 보안 연결로만 받아옵니다
- 어떤 버전의 의존성을 쓸지 의도적으로 결정하고 필요할 때만 업그레이드 해야합니다
- 유지보수되지 않거나 오래된 버전에 보안 패치를 제공하지 않는 라이브러리를 모닡터링 한다
- CI/CD IDE 개발도구를 정기적으로 업데이트 합니다
- 업데이트를 모든 시스템에 한 번에 배포하지 말고 단계적으로나 카나리 배포를 써서 피해를 줄입니다

# Security Misconfiguration
소프트웨어가 점점 더 많은 설정 옵션을 가지게 되면서 이 항목이 올라오게 되었습니다
보안 설정 오류는 시스템, 애플리케이션, 또는 클라우드 서비스가 보안 관점에서 잘못 설정되어 취약점이 생기는 상태를 말합니다

### 취약점

- 앱이 스택 전반에 걸친 적절한 보안 하드닝이 없거나 클라우드 서비스 권한이 잘못 설정된 경우
- 불필요한 기능이 활성화되거나 설치된 경우 (포트, 서비스, 페이지, 계정 등)
- 기본 계정과 기본 비밀번호가 그대로 살아있는 경우
- 과도한 오류 메시지를 가로채는 중앙설정이 없는 경우
- 업그레이드된 시스템에서 최신 보안 기능이 비활성화 되어있거나 안전하지 않게 설정된 경우
- 하위 호환성을 너무 우선시한 나머지 안전하지 않은 설정이 유지되는 경우
- 애플리케이션 서버, 프레임워크, 라이브러리, DB 등의 보안 설정 값이 안전한 값으로 맞춰져 있지 않은 경우
- 서버가 보안 헤더나 보안 지시문을 보내지 않거나 보내더라도 안전하지 않은 값으로 설정된 경우

### 방지방법

- 반복 가능한 프로세스를 만듭니다
새 환경을 빠르고 쉽게 잠금이 잘 된 상태로 배포할 수 있어야 합니다
개발, QA, 운영 환경은 구성은 동일하게 맞추고 자격 증명만 다르게 써야 합니다 ← 자동화가 좋음
- 최소 플랫폼만 운영합니다
불필요한 기능, 컴포넌트, 문서, 샘플은 제거하거나 아예 설치하지 않습니다
- 패치 관리 프로세스의 일부로 보안 공지, 업데이트 패치에 맞게 설정을 검토 수정합니다
클라우드 저장소 권한, 예를 들면 S3 버킷 권한도 점검해야 합니다
- 분리된 애플리케이션 아키텍쳐를 사용합니다
컴포넌트나 테넌트 사이클 세그먼트, 컨테이너, 클라우드 보안 그룹 등로 효과적으로 분리해야합니다
- 클라이언트에 보안 지시문 예를 들면 보안 헤더를 보냅니다
- 모든 환경에서 설정이 제대로 먹히는지 자동 검증 프로세스를 둡니다
- 과도한 오류 메시지를 막기 위해 중앙 집중식 예외 처리 설정을 선제적으로 넣습니다
- 자동 검증이 없다면 최소한 매년 수동 검증이라도 해야합니다
- 코드, 설정 파일, 파이프라인에 정적 키나 시크릿을 박아 넣지 말고, 플랫폼이 제공하는 아이덴티티 연합, 단기 자격 증명, 역할 기반 접근 방식을 사용합니다
